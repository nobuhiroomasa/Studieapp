# 勉強くん フォルダ構成

## 1. このファイルの目的

このドキュメントは、iPhoneアプリ「勉強くん」を SwiftUI / SwiftData で実装するための推奨フォルダ構成を定義する。

「画面デザイン画像12枚」「画面設計書画像13枚（一覧プレビュー1枚＋各画面設計書12枚）」「画面仕様書」をもとに、Codex が迷わず実装できるように、画面・モデル・共通部品・サービスの置き場所を明確にする。

---

## 2. 技術前提

| 項目 | 内容 |
|---|---|
| 対象 | iPhoneアプリ |
| UI | SwiftUI |
| データ保存 | SwiftData |
| ログイン | 初期版では不要 |
| 通信 | 初期版では不要 |
| 対象試験 | 第二種電気工事士 筆記試験 |
| 問題形式 | 4択固定 |
| 初期データ | 少数のサンプル問題 |

## 4. 各フォルダの役割

## 4-1. App

アプリ全体の起動、タブ、ルーティング、共通環境を管理する。

| ファイル | 役割 |
|---|---|
| `RootView.swift` | 初期表示。必要なら初期データ投入を実行する |
| `MainTabView.swift` | ホーム・問題演習・復習・記録・設定の下部タブを管理する |
| `AppRoute.swift` | 画面遷移先を enum で管理する |
| `AppEnvironment.swift` | アプリ全体で使う定数や依存をまとめる |

---

## 4-2. Models

SwiftData の `@Model` を配置する。

初期版で必須のモデルは以下。

| モデル | 内容 |
|---|---|
| `Question` | 問題データ |
| `AnswerHistory` | 回答履歴 |
| `ReviewStatus` | 復習状態 |
| `StudyGoal` | 目標設定 |
| `AppSetting` | 初期データ投入済みフラグなど |

### Enums

SwiftData では enum をそのまま永続化するより、`String` や `Int` の raw value として保存する方が扱いやすい。

例：

```swift
var categoryRaw: String
var correctOptionRaw: String
```

View 側では computed property で enum に変換する。

---

## 4-3. Repositories

SwiftData への保存・取得・削除処理をまとめる。

View から直接 SwiftData の細かいクエリを書きすぎると、画面が重くなる。  
そのため、データ操作は Repository に寄せる。

| Repository | 役割 |
|---|---|
| `QuestionRepository` | 問題の取得・作成・更新・論理削除 |
| `AnswerHistoryRepository` | 回答履歴の保存・集計用取得 |
| `ReviewRepository` | 復習対象の登録・解除・取得 |
| `StudyGoalRepository` | 目標設定の取得・保存 |

---

## 4-4. Services

複数モデルをまたぐロジックや集計処理を配置する。

| Service | 役割 |
|---|---|
| `QuizService` | 出題条件に合う問題を抽出する |
| `StudyStatsService` | 正答率、回答数、連続学習日数などを集計する |
| `ReviewService` | 不正解時の復習登録、2回連続正解時の解除を行う |
| `GoalProgressService` | 今日の目標達成率、試験日までの日数を計算する |
| `SampleDataSeeder` | 初回起動時にサンプル問題を登録する |

---

## 4-5. ViewModels

画面ごとの状態管理と表示用データ作成を行う。

| ViewModel | 対応画面 |
|---|---|
| `HomeViewModel` | ホーム画面 |
| `QuizSettingViewModel` | 問題演習設定画面 |
| `QuizSessionViewModel` | 問題演習、回答結果、演習完了 |
| `ReviewViewModel` | 復習画面 |
| `QuestionFormViewModel` | 問題追加・編集フォーム |
| `QuestionListViewModel` | 問題一覧画面 |
| `StudyRecordViewModel` | 学習記録画面 |
| `GoalSettingViewModel` | 目標設定画面 |
| `SettingsViewModel` | 設定画面 |

---

## 4-6. Views

画面単位で SwiftUI の View を配置する。

ルール：

- 1画面1フォルダを基本にする
- 画面専用の小さい部品は同じフォルダ内に置いてもよい
- 複数画面で使う部品は `Components/` に移動する
- 回答結果画面は正解・不正解で別ファイルにしない

回答結果画面は以下のように状態で出し分ける。

```swift
AnswerResultView(isCorrect: Bool)
```

---

## 4-9. Docs

設計資料と画面画像を入れる。

Codex に実装させる場合、画像資料をこのフォルダに配置しておくと参照しやすい。

注意：`Docs/` 配下は開発資料であり、アプリ本体の画面素材ではない。Xcodeに追加する場合も、原則としてアプリターゲットには含めない。`Assets.xcassets` に入れるのは、アプリ内で実際に表示する画像・アイコンのみとする。

推奨：

```text
Docs/ScreenDesigns/      画面デザイン画像12枚
Docs/ScreenSpecs/        画面設計書画像13枚（一覧プレビュー1枚 + 各画面設計書12枚）
Docs/MD/                 今回作成したMDファイル
```

## 6. 実装上の注意

1. 初期版ではログインを作らない。
2. すべて端末内保存にする。
3. 問題は4択固定にする。
4. 回答後は即時に結果画面へ遷移する。
5. 正解画面・不正解画面は `AnswerResultView` に統合する。
6. 復習対象は `ReviewStatus` で管理する。
7. 問題削除は物理削除ではなく論理削除にする。
8. 下部タブは5つに固定する。
9. 問題演習中は集中のため下部タブを非表示にしてよい。
10. 画面デザイン画像と違うUIにしすぎない。
